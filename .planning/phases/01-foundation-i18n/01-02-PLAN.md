# Plan 02: i18n Infrastructure & Locale Routing

## Objective

Set up the complete next-intl i18n infrastructure: routing config, proxy middleware, request config, navigation helpers, message files, and the `[locale]` dynamic segment with locale-scoped layout. After this plan, navigating to `/en` and `/bg` renders locale-specific content with correct `<html lang>` attributes, and the root `/` redirects to `/en`.

Purpose: i18n routing is the architectural skeleton for the entire site. Every page, every component, every link must be locale-aware. Getting this right in Phase 1 prevents painful rewrites later.

Output: Working i18n routing where `/en` shows English content, `/bg` shows Bulgarian placeholder content, and `/` redirects to the default locale.

## Context

@.planning/phases/01-foundation-i18n/01-RESEARCH.md — Complete code patterns for every file in this plan (Patterns 1-7, 9-10).

**Critical anti-patterns from research:**
- NEVER use `middleware.ts` -- Next.js 16 uses `proxy.ts`
- NEVER access `params` synchronously -- always `await params` (it is a Promise in Next.js 16)
- NEVER skip `setRequestLocale(locale)` -- required for static rendering
- NEVER skip `generateStaticParams` in `[locale]/layout.tsx` -- required for build-time static rendering
- NEVER put `"use client"` in layout.tsx or page.tsx

**Open questions from research (handle during implementation):**
1. `proxy.ts` import path: Try `@/i18n/routing` first. If it fails (because proxy.ts is outside src/), fall back to `./src/i18n/routing`.
2. Root layout: The `<html>` and `<body>` tags go in `src/app/[locale]/layout.tsx` (not root layout) so `lang` can be dynamic. Root `src/app/layout.tsx` just passes through children.

import {getRequestConfig} from 'next-intl/server';

export default getRequestConfig(async ({requestLocale}) => {
  const requested = await requestLocale;
  const locale = hasLocale(routing.locales, requested)
    ? requested
    : routing.defaultLocale;

  return {
    locale,
    messages: (await import(`../../messages/${locale}.json`)).default

});

**File 1: `proxy.ts`** (at PROJECT ROOT, not inside src/)

import createMiddleware from 'next-intl/middleware';

IMPORTANT: The import path MUST be `'./src/i18n/routing'` (relative from project root), NOT `'@/i18n/routing'`. The `@/` alias maps to `./src/*` but `proxy.ts` sits outside `src/`, so the alias resolution may not work in the middleware/proxy context. Use the relative path to be safe.

<task id="3" depends_on="2">
**Create locale layout and home page with translations and Cyrillic font**

**Step 1: Update root layout `src/app/layout.tsx`**

The root layout must become a minimal pass-through. It must NOT include `<html>` or `<body>` tags because those will be in the locale layout where `lang` can be set dynamically.

```typescript
export default function RootLayout({
  children,
}: Readonly<{
  children: React.ReactNode;
}>) {
  return children;
}
```

That is the ENTIRE file. No `<html>`, no `<body>`, no metadata export. Just pass through children. The locale layout handles everything.

**Step 2: Create `src/app/[locale]/layout.tsx`**

```typescript
import {NextIntlClientProvider, hasLocale} from 'next-intl';
import {notFound} from 'next/navigation';
import {routing} from '@/i18n/routing';
import {setRequestLocale} from 'next-intl/server';
import {Inter} from 'next/font/google';
import '../globals.css';

const inter = Inter({
  subsets: ['latin', 'cyrillic'],
  display: 'swap',
  variable: '--font-inter',
});

type Props = {
  children: React.ReactNode;
  params: Promise<{locale: string}>;
};

export function generateStaticParams() {
  return routing.locales.map((locale) => ({locale}));
}

export default async function LocaleLayout({children, params}: Props) {
  const {locale} = await params;
  if (!hasLocale(routing.locales, locale)) {
    notFound();
  }

  setRequestLocale(locale);

  return (
    <html lang={locale} className={inter.variable}>
      <body className="font-sans antialiased">
        <NextIntlClientProvider>
          {children}
        </NextIntlClientProvider>
      </body>
    </html>
  );
}
```

Key points:
- `params` is awaited (Promise in Next.js 16)
- `setRequestLocale` called for static rendering
- `generateStaticParams` returns both locales for build-time rendering
- Inter font with `subsets: ['latin', 'cyrillic']` for Cyrillic support
- `display: 'swap'` prevents flash of invisible text
- `--font-inter` CSS variable integrates with Tailwind via `--font-sans`
- `globals.css` imported HERE (in the locale layout), not in root layout
- NO `"use client"` directive

**Step 3: Create `src/app/[locale]/page.tsx`**

```typescript
import {useTranslations} from 'next-intl';
import {setRequestLocale} from 'next-intl/server';
import {getTranslations} from 'next-intl/server';

type Props = {
  params: Promise<{locale: string}>;
};

export async function generateMetadata({params}: Props) {
  const {locale} = await params;
  const t = await getTranslations({locale, namespace: 'Metadata'});

  return {
    title: t('title'),
    description: t('description'),
  };
}

export default async function HomePage({params}: Props) {
  const {locale} = await params;
  setRequestLocale(locale);

  // useTranslations is valid in Server Components that call setRequestLocale
  const t = useTranslations('HomePage');

  return (
    <main className="flex min-h-screen flex-col items-center justify-center p-24">
      <h1 className="text-4xl font-bold tracking-tight">
        {t('title')}
      </h1>
      <p className="mt-4 text-lg text-muted-foreground max-w-2xl text-center">
        {t('description')}
      </p>
      <div className="mt-8 flex gap-4">
        <button className="rounded-md bg-primary px-6 py-3 text-primary-foreground font-medium hover:bg-primary/90">
          {t('cta')}
        </button>
        <button className="rounded-md border border-input px-6 py-3 font-medium hover:bg-accent hover:text-accent-foreground">
          {t('ctaSecondary')}
        </button>
      </div>
    </main>
  );
}
```

Key points:
- `params` is awaited (Promise in Next.js 16)
- `setRequestLocale` called before `useTranslations`
- `generateMetadata` provides locale-aware page title/description for SEO
- NO `"use client"` directive -- this is a server component
- Uses Tailwind classes for minimal styling
