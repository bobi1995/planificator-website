## Objective

Build the complete MDX blog infrastructure that all subsequent blog work depends on. This includes installing packages, creating the content registry that reads MDX files from disk with gray-matter for frontmatter parsing, setting up Tailwind typography plugin for prose styling, creating the MDX component map for custom rendering, and seeding 2 blog posts per locale (4 total) to validate the pipeline end-to-end.

**Approach:** Use `next-mdx-remote/rsc` to compile and render MDX from raw strings on the server. This avoids the `@next/mdx` bundler pipeline entirely -- no `next.config.ts` changes, no `pageExtensions`, no dynamic `import()` path resolution issues. Content is read from disk with `fs.readFile`, frontmatter parsed with `gray-matter`, and MDX compiled at request/build time by `next-mdx-remote`.

**Step 1: Create the content registry at `src/lib/blog.ts`**

This is the core utility that reads MDX files from disk, parses frontmatter, and exposes query functions. All functions are wrapped with React `cache()` for request-level deduplication.

```typescript
import fs from 'fs/promises';
import path from 'path';
import matter from 'gray-matter';
import { cache } from 'react';

export interface BlogPostMeta {
  title: string;
  description: string;
  date: string;
  author: string;
  tags: string[];
  image?: string;
  published?: boolean;
}

export interface BlogPost {
  slug: string;
  meta: BlogPostMeta;
  content: string; // raw MDX content (without frontmatter)
}

const CONTENT_DIR = path.join(process.cwd(), 'src', 'content', 'blog');

/**
 * Estimated reading time in minutes (average 200 words/min).
 */
export function getReadingTime(content: string): number {
  const words = content.trim().split(/\s+/).length;
  return Math.max(1, Math.ceil(words / 200));
}

/**
 * Returns all published posts for a locale, sorted by date descending.
 */
export const getAllPosts = cache(async (locale: string): Promise<BlogPost[]> => {
  const dir = path.join(CONTENT_DIR, locale);

  let files: string[];
  try {
    files = await fs.readdir(dir);
  } catch {
    return []; // locale directory doesn't exist yet
  }

  const posts = await Promise.all(
    files
      .filter((f) => f.endsWith('.mdx'))
      .map(async (file) => {
        const raw = await fs.readFile(path.join(dir, file), 'utf-8');
        const { data, content } = matter(raw);
        return {
          slug: file.replace(/\.mdx$/, ''),
          meta: data as BlogPostMeta,
          content,
        };
      })
  );

  return posts
    .filter((p) => p.meta.published !== false)
    .sort(
      (a, b) =>
        new Date(b.meta.date).getTime() - new Date(a.meta.date).getTime()
    );
});

/**
 * Returns a single post by slug, or null if not found.
 */
export const getPostBySlug = cache(
  async (locale: string, slug: string): Promise<BlogPost | null> => {
    const filePath = path.join(CONTENT_DIR, locale, `${slug}.mdx`);
    try {
      const raw = await fs.readFile(filePath, 'utf-8');
      const { data, content } = matter(raw);
      const meta = data as BlogPostMeta;
      if (meta.published === false) return null;
      return { slug, meta, content };
    } catch {
      return null;
    }
  }
);

/**
 * Returns posts filtered by tag for a locale.
 */
export const getPostsByTag = cache(
  async (locale: string, tag: string): Promise<BlogPost[]> => {
    const posts = await getAllPosts(locale);
    return posts.filter((p) =>
      p.meta.tags.map((t) => t.toLowerCase()).includes(tag.toLowerCase())
    );
  }
);

/**
 * Returns all unique tags across all posts for a locale.
 */
export const getAllTags = cache(async (locale: string): Promise<string[]> => {
  const posts = await getAllPosts(locale);
  const tagSet = new Set<string>();
  for (const post of posts) {
    for (const tag of post.meta.tags) {
      tagSet.add(tag);
    }
  }
  return Array.from(tagSet).sort();
});

/**
 * Returns all valid slugs for a locale (for generateStaticParams).
 */
export const getPostSlugs = cache(async (locale: string): Promise<string[]> => {
  const posts = await getAllPosts(locale);
  return posts.map((p) => p.slug);
});
```

Key implementation details:
- `cache()` from React ensures each function is called only once per request, even if invoked in both `generateMetadata` and page component.
- `CONTENT_DIR` uses `process.cwd()` for absolute paths -- works in both dev and build.
- `getAllPosts` handles missing locale directories gracefully (returns empty array).
- `getPostBySlug` reads a single file directly for efficiency.
- `getReadingTime` provides estimated read time for PostCard display.

**Step 2: Create MDX component map at `src/components/mdx/mdx-components.tsx`**

This file defines custom React components that replace standard HTML elements when rendering MDX content. It also includes any custom components (like Callout) that can be used inside MDX files.

```typescript
import type { MDXComponents } from 'mdx/types';
import { Callout } from './Callout';

export const mdxComponents: MDXComponents = {
  // Custom components available in MDX files
  Callout,

  // Override heading elements for brand typography + slug anchors
  h1: ({ children, id, ...props }) => (
    <h1 id={id} className="text-display mb-6 scroll-mt-20" {...props}>
      {children}
    </h1>
  ),
  h2: ({ children, id, ...props }) => (
    <h2 id={id} className="text-heading mt-12 mb-4 scroll-mt-20" {...props}>
      {children}
    </h2>
  ),
  h3: ({ children, id, ...props }) => (
    <h3 id={id} className="text-subheading mt-8 mb-3 scroll-mt-20" {...props}>
      {children}
    </h3>
  ),
};
```

Note: We do NOT override `p`, `a`, `ul`, `li`, etc. because the `prose` class from `@tailwindcss/typography` handles their styling. We only override headings to use the brand typography scale and add `scroll-mt-20` for proper scroll-to-anchor offset (below the sticky header).

**Step 3: Create Callout component at `src/components/mdx/Callout.tsx`**

A reusable callout/tip/warning box for MDX content.

```typescript
import { cn } from '@/lib/utils';

type CalloutType = 'info' | 'warning' | 'tip';

const calloutStyles: Record<CalloutType, string> = {
  info: 'border-brand-300 bg-brand-50 text-brand-900',
  warning: 'border-yellow-300 bg-yellow-50 text-yellow-900',
  tip: 'border-green-300 bg-green-50 text-green-900',
};

const calloutLabels: Record<CalloutType, string> = {
  info: 'Info',
  warning: 'Warning',
  tip: 'Tip',
};

interface CalloutProps {
  type?: CalloutType;
  title?: string;
  children: React.ReactNode;
}

export function Callout({ type = 'info', title, children }: CalloutProps) {
  return (
    <div
      className={cn(
        'my-6 rounded-lg border-l-4 p-4 not-prose',
        calloutStyles[type]
      )}
    >
      <p className="mb-1 text-sm font-semibold">{title || calloutLabels[type]}</p>
      <div className="text-sm">{children}</div>
    </div>
  );
}
```

Key details:
- `not-prose` class removes Tailwind typography styling inside the callout (the callout has its own styling).
- Three variants: `info` (brand blue), `warning` (yellow), `tip` (green).
- Used in MDX files as `<Callout type="tip">...</Callout>`.

# The Complete Guide to Production Scheduling

Production scheduling is the backbone of efficient manufacturing. It determines which orders run on which machines, when they start, and how resources are allocated across shifts.

## Why Production Scheduling Matters

Without a clear production schedule, manufacturers face:

- **Missed deadlines** -- orders compete for resources without clear priority
- **Wasted capacity** -- machines sit idle while other resources are overloaded
- **Excessive changeovers** -- poor sequencing leads to unnecessary setup time
- **Firefighting culture** -- planners spend more time reacting than planning

A well-optimized schedule addresses all of these problems simultaneously.

## From Spreadsheets to AI

Traditional scheduling relies on spreadsheets and tribal knowledge. A planner builds the weekly schedule manually, balancing constraints in their head. This works for small operations, but breaks down as complexity grows.

Modern scheduling tools like Planifactor use **AI optimization** to evaluate thousands of possible schedules and find the best one automatically. The planner defines objectives (minimize late orders, reduce changeover time, balance workload) and the optimizer does the heavy lifting.

<Callout type="tip">
AI scheduling does not replace the planner -- it gives them superpowers. The planner still makes the strategic decisions; the AI handles the combinatorial complexity.
</Callout>

## Key Components of a Good Schedule

### 1. Gantt Chart Visualization

A Gantt chart is the standard way to visualize a production schedule. Each row represents a resource (machine, worker, station), and bars represent operations placed on a timeline. A good Gantt chart lets you:

- See the entire production timeline at a glance
- Identify bottlenecks and idle time visually
- Drag and drop operations to adjust the schedule

### 2. Resource Management

Every schedule depends on knowing what resources are available. This includes machines, workers, workstations, and tooling. Each resource has capacity limits, shift patterns, and maintenance windows.

### 3. Constraint Handling

Real production involves hard constraints (machine availability, order deadlines) and soft constraints (preferred sequences, load balancing). A good scheduler respects all of them.

## Getting Started

The first step is mapping your current process: what resources do you have, what orders are in the pipeline, and what are your scheduling objectives. From there, tools like Planifactor can generate an optimized schedule in minutes.

# Пълно ръководство за производствено планиране

Производственото планиране е основата на ефективното производство. То определя кои поръчки работят на кои машини, кога започват и как ресурсите се разпределят между смените.

## Защо производственото планиране е важно

Без ясен производствен график, производителите се сблъскват с:

- **Пропуснати срокове** -- поръчките се конкурират за ресурси без ясен приоритет
- **Пропиляна мощност** -- машините стоят без работа, докато другите ресурси са претоварени
- **Прекомерни пренастройки** -- лошото секвениране води до ненужно време за настройка
- **Култура на пожари** -- планиращите прекарват повече време в реакция, отколкото в планиране

Добре оптимизираният график решава всички тези проблеми едновременно.

## От таблици към AI

Традиционното планиране разчита на електронни таблици и опита на хората. Планиращият изгражда седмичния график ръчно, балансирайки ограниченията наум. Това работи за малки операции, но се срива с нарастването на сложността.

Модерните инструменти за планиране като Planifactor използват **AI оптимизация** за оценка на хиляди възможни графици и автоматично намиране на най-добрия. Планиращият определя целите (минимизиране на закъснелите поръчки, намаляване на времето за пренастройка, балансиране на натоварването), а оптимизаторът върши тежката работа.

<Callout type="tip">
AI планирането не заменя планиращия -- то му дава суперсили. Планиращият все още взема стратегическите решения; AI се справя с комбинаторната сложност.
